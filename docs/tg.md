# 🤖 Telegram-бот системы записи — согласованная концепция

## 1. Назначение бота

Telegram-бот — **единый интерфейс управления и доступа** к системе записи в Telegram.

Бот:
- **не содержит бизнес-логики**
- **не работает напрямую с БД**
- работает **исключительно через FastAPI**
- обслуживает **одну компанию**
- маршрутизирует пользователей по ролям

---

## 2. Базовые ограничения

- **Один бот → одна компания**
- `company_id`:
  - передаётся боту при инициализации (env / config) или id записывается единожды при создании компании. Последующее изменение только в ручную.
  - не выбирается и не меняется через интерфейс
- Админы существуют **только в этом боте**
- Redis используется backend’ом и ботом (бот — только для UI-состояния, не для бизнес-логики)
- Бот работает строго по контракту `API.md`

## 2.1. Инициализация системы (bootstrap)

При первом запуске система может находиться в состоянии:
- пустая база данных
- отсутствие пользователей
- отсутствие администраторов

Для детерминированного запуска используется bootstrap-инициализация.

Принцип:
- `ADMIN_TG_ID` задаётся через `.env`
- при запуске backend выполняется инициализационный скрипт
- бот в инициализации не участвует
- Redis не используется

Алгоритм bootstrap:
1. Считать `ADMIN_TG_ID` из `.env`
2. Проверить наличие пользователей в БД
3. Если пользователей нет:
   - создать компанию (если отсутствует)
   - создать первого пользователя
   - назначить ему роль `admin`
4. Если пользователи есть:
   - проверить, существует ли admin с данным `tg_id`
   - если нет — создать нового admin
   - существующие админы получают уведомление (через backend / bot)

Инициализация:
- выполняется скриптом
- идемпотентна
- не выполняется в runtime

## 2.2. Audit и bootstrap-события

Все критические события инициализации и управления ролями
фиксируются в `audit_log`.

Audit используется для:
- отслеживания bootstrap-инициализации
- фиксации добавления администраторов
- уведомления существующих администраторов
- последующего анализа и безопасности

Telegram-бот:
- не пишет audit напрямую
- отображает audit-события через backend API

## 3. Роли и сценарии

### 3.1 Admin

**Назначение**
- наполнение справочников
- управление специалистами
- управление связями
- управление финансами
- служебные операции

**Авторизация**
- по `tg_id`
- первичный `tg_id` администратора задаётся через `.env`
- привязка выполняется bootstrap-скриптом
- дальнейшее управление ролями — только через API
- смена `tg_id` через API запрещена

**Функции**
- locations / rooms / services / service_packages
- specialists (профиль, график)
- specialist ↔ services (domain API)
- calendar_overrides
- кошелёк клиента — **только domain wallet API**
- просмотр / служебное создание бронирований

---

### 3.2 Specialist

**Назначение**
- чтение своих записей

**Авторизация**
- только по `tg_id`
- жёсткая связь `tg_id → users.id`
- смена `tg_id` запрещена

**Функции (v1)**
- просмотр своих booking
- просмотр расписания

---

### 3.3 Client

**Назначение**
- просмотр услуг и записей
- запись через **Mini App** (WebApp) или **сайт** (web booking)

**Входы**
- Web (сайт): имя + телефон
- Telegram / Mini App: `tg_id`

**Клавиатура (ReplyKeyboard)**
```
[ Мои записи | Услуги ]
[      Связаться      ]
```

Кнопка «Записаться» в боте отключена — запись через Mini App (WebApp button) или сайт.

**Функции**
- просмотр записей
- просмотр услуг и цен
- связь с поддержкой
- запись через Mini App или сайт (phone gate: soft, non-blocking)

## 4. Определение роли при входе

1. Пользователь пишет боту
2. Бот получает `tg_id`
3. API:
   - ищет пользователя по `tg_id`
4. Если не найден:
   - создаётся пользователь с ролью `client`
5. Роль определяется через `user_roles`
6. Бот запускает сценарий:
   - admin / specialist / client

Примечание:
- bootstrap-инициализация выполняется ДО первого входа в бот
- бот никогда не создаёт администратора самостоятельно
- бот всегда работает с уже инициализированными пользователями

## 4.1 Аутентификация через gateway

Аутентификация выполняется в gateway, НЕ в боте.

Поток:
1. Telegram → gateway `/tg/webhook`
2. gateway: extract_tg_id(update)
3. gateway: проверка кэша Redis `tg_user:{tg_id}` (TTL 10 мин)
4. Если нет в кэше → запрос к backend:
   - GET /users/by_tg/{tg_id}
   - Если 404 → создать пользователя + роль client
5. gateway передаёт в bot: TgUserContext(tg_id, user_id, company_id, role)
6. bot использует role синхронно (без HTTP-запросов)

Бот:
- НЕ делает запросы к backend для аутентификации
- получает готовый контекст от gateway

## 5. Работа с бронированиями

### Принцип
- **Открытая запись**
- Бот **не рассчитывает слоты**
- Бот **не работает с Redis**

### Реализация
- `POST /bookings` существует
- внутри backend:
  - Redis-lock
  - повторная проверка слотов
- без Redis создание брони невозможно

Бот:
- вызывает API
- обрабатывает результат
- не знает о механике блокировок

---

## 6. Связь Specialist ↔ Services

- Тип связи: **M:N**
- Таблица: `specialist_services`
- Отдельного CRUD-ресурса **нет**
- Используются доменные маршруты:

```
GET /specialists/{id}/services
POST /specialists/{id}/services
PATCH /specialists/{id}/services/{service_id}
DELETE /specialists/{id}/services/{service_id}
```


- Soft-delete через `is_active`

---

## 7. Финансы (кошелёк)

- CRUD кошелька запрещён
- Все операции **только через domain API**:

```
GET /wallets/{user_id}
GET /wallets/{user_id}/transactions
POST /wallets/{user_id}/deposit
POST /wallets/{user_id}/withdraw
POST /wallets/{user_id}/payment
POST /wallets/{user_id}/refund
POST /wallets/{user_id}/correction
POST /wallets/{user_id}/package-purchase
POST /wallets/{user_id}/package-refund
```

Бот:
- не изменяет баланс напрямую
- не работает с `wallet_transactions`

---

## 8. Графики и пересчёт слотов

- После наполнения справочников:
  - backend автоматически считает слоты на **60 дней**
- Через бота:
  - **только корректировки** (overrides)
- Любое изменение:
  - триггерит пересчёт на backend
- Redis — runtime слой, бот его не трогает

---

## 9. Принципиальные запреты

Бот **никогда не**:
- обращается к БД напрямую
- считает слоты
- меняет баланс через CRUD
- меняет `tg_id`
- обходит API.md

## 10. Итог

bootstrap и начальная инициализация выполняются вне бота (скриптом)

Telegram-бот:
- UI + авторизация + маршрутизация
- строгий клиент FastAPI
- одна компания
- три роли
- нулевая бизнес-логика

## 11. Навигация и клавиатуры

> **Полный UI-контракт клавиатур:** [`docs/tg_kbrd.md`](tg_kbrd.md) (v3.0)
>
> Там описаны: типы переходов (A/B1/B2/C), MenuController, якорь меню,
> inline cleanup, пагинация, Redis-хранилище, чеклист и запреты.

Краткие принципы:

- **MenuController** (`bot/app/utils/menucontroller.py`) — единственный способ управления клавиатурами
- Навигация через reply-handlers (`handlers/*_reply.py`), не через FSM
- FSM — только для бизнес-сценариев (wizard, ввод данных)
- `last_menu_message_id` хранится в Redis (`tg:menu:{chat_id}`)
- `ReplyKeyboardRemove` запрещён между меню
- Текст кнопок локализован через i18n

## 12. Структура handlers

```
bot/
├── main.py                    # dp, bot, process_update, route_by_role
├── keyboards/                 # структура кнопок (is_persistent=True)
├── flows/                     # FSM-сценарии по ролям
├── handlers/                  # reply-роутинг: text → flow action
└── utils/menucontroller.py    # транспорт: send/delete, Redis
```

- Один router на роль (`admin_reply.py`, `specialist_reply.py`, `client_reply.py`)
- `route_by_role()` — только точка входа (`/start`), дальше навигация через reply-handlers
